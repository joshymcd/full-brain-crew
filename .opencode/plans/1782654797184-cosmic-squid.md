# Plan: Map OpenCode Chat Log Data To Personal Assistant UI

## Goal

Plan how to render real OpenCode conversation data in the `/chats/$chatId` page. The current route already fetches the chat log with TanStack Query, but still renders `dummyRows`. This plan inventories the data returned by OpenCode, evaluates what is useful for a personal-assistant-focused UI, and maps each data type to existing shadcn chat primitives.

## Current State

- Route file: `webapp/src/routes/chats/$chatId.tsx`
- Current query:

  ```ts
  const chatLogQuery = useQuery({
    queryKey: ["opencode", "chat-log", opencodeDirectory, chatId],
    queryFn: async () =>
      (
        await opencodeClient.session.messages({
          ...locationQuery,
          sessionID: chatId,
        })
      ).data,
    retry: false,
  })
  ```

- Current UI ignores the query with `void chatLogQuery` and renders static `dummyRows`.
- URL uses the OpenCode `session.id`, not slug, so `session.messages({ sessionID: chatId })` can fetch directly without a `session.list()` lookup.
- Current SDK endpoint returns the legacy message shape:

  ```ts
  Array<{
    info: Message
    parts: Array<Part>
  }>
  ```

- SDK source references:
  - `SessionMessagesData` and response shape: `@opencode-ai/sdk/dist/v2/gen/types.gen.d.ts:6655-6688`
  - `UserMessage`, `AssistantMessage`, `Part`: `@opencode-ai/sdk/dist/v2/gen/types.gen.d.ts:128-474`

## Available shadcn UI Primitives

- `Message`, `MessageAvatar`, `MessageContent`, `MessageHeader`, `MessageFooter`
  - Use for complete transcript rows, alignment, sender metadata, model/agent/cost metadata.
- `Bubble`, `BubbleContent`, `BubbleReactions`
  - Use for readable message bodies and rich inline content.
  - Variants to use:
    - `default`: user messages
    - `secondary`: assistant messages
    - `muted`: quiet metadata or low-emphasis assistant context
    - `outline`: tool/patch/detail cards
    - `ghost`: reasoning or transparent assistant prose
    - `destructive`: errors
- `Marker`, `MarkerIcon`, `MarkerContent`
  - Use for status rows, system events, step boundaries, retries, compaction, agent switches, separators.
- `Attachment`, `AttachmentGroup`, `AttachmentMedia`, `AttachmentContent`, `AttachmentTitle`, `AttachmentDescription`, `AttachmentActions`, `AttachmentTrigger`
  - Use for prompt files, referenced files, generated artifacts, tool attachments, patch files.
- `MessageScrollerProvider`, `MessageScroller`, `MessageScrollerViewport`, `MessageScrollerContent`, `MessageScrollerItem`, `MessageScrollerButton`
  - Keep for transcript frame, scroll anchoring, and jump-to-latest behavior.

## Data Inventory And UI Mapping

### Message Envelopes

#### `UserMessage`

Fields:
- `id`, `sessionID`, `role: "user"`
- `time.created`
- `format?`
- `summary?` with optional title/body/diffs
- `agent`
- `model`
- `system?`
- `tools?`

Usefulness:
- High. This is the human prompt and should be the main right-aligned user turn.
- `time.created` is useful in `MessageHeader` or `MessageFooter`.
- `agent`, `model`, and `tools` are useful as optional metadata, not primary content.
- `summary.diffs` can be useful for showing what the user request changed, but should be secondary.

UI mapping:
- `Message align="end"`
- `MessageAvatar` with user initials or profile avatar
- `MessageHeader`: "You" plus formatted time if desired
- `Bubble variant="default" align="end"`
- `BubbleContent`: visible `text` parts
- `AttachmentGroup`: any `file` parts associated with the user prompt
- `MessageFooter`: optional agent/model/tools metadata if enabled

Conditions:
- If no text exists but files exist, render a small bubble such as "Attached files" plus attachments.
- If `summary.diffs` exists, render a compact `Marker` or patch attachment after the user message.

#### `AssistantMessage`

Fields:
- `id`, `sessionID`, `role: "assistant"`
- `time.created`, `time.completed?`
- `error?`
- `parentID`
- `modelID`, `providerID`, `variant?`
- `mode`, `agent`
- `path.cwd`, `path.root`
- `summary?`
- `cost`, `tokens`
- `structured?`
- `finish?`

Usefulness:
- High. This is the assistant answer and activity container.
- `error` is high priority and should be visible.
- `model`, `agent`, `mode`, `finish`, cost, and tokens are useful but should be secondary.
- `path` is usually developer/debug metadata; hide by default in a personal assistant UI.

UI mapping:
- `Message align="start"`
- `MessageAvatar` with OpenCode/assistant initials
- `MessageHeader`: assistant name, optional agent/model
- `Bubble variant="secondary"` for text answer
- `Bubble variant="destructive"` if `info.error` exists
- `MessageFooter`: time, finish reason, model/provider, compact token/cost summary if enabled

Conditions:
- If assistant has only tool/status parts and no text, render the assistant row as markers/tool cards rather than an empty bubble.
- If `error` exists, render the error message prominently and keep tool/status context below it.

### Part Types

#### `TextPart`

Fields:
- `type: "text"`
- `text`
- `synthetic?`, `ignored?`
- `time?`
- `metadata?`

Usefulness:
- Highest. This is the primary readable conversation content.

UI mapping:
- `BubbleContent` paragraphs inside the parent `Message`.

Conditions:
- Skip or visually de-emphasize `ignored` text.
- If `synthetic` is true, consider `Marker` or `Bubble variant="muted"` instead of primary text.
- Multiple text parts can become separate paragraphs in one bubble.

#### `ReasoningPart`

Fields:
- `type: "reasoning"`
- `text`
- `metadata?`
- `time.start`, `time.end?`

Usefulness:
- Medium to high. Useful for transparency, but can overwhelm a personal-assistant UI.

UI mapping:
- Default: collapsed or low-emphasis section under assistant answer.
- `Bubble variant="ghost"` or `Bubble variant="muted"` labelled "Reasoning".
- Future enhancement: use a collapsible shadcn component if added.

Conditions:
- Show only when non-empty.
- Collapse or summarize if long.
- Hide by default if the UI should feel consumer/personal rather than developer-focused.

#### `FilePart`

Fields:
- `type: "file"`
- `mime`, `filename?`, `url`
- `source?`, which may be:
  - file source: `path`, text range
  - symbol source: `path`, range, name, kind
  - resource source: `clientName`, `uri`, text

Usefulness:
- High. Files are important for assistant context and artifacts.

UI mapping:
- `AttachmentGroup`
- `Attachment`
- `AttachmentMedia` with file/code/image icon based on `mime` or path
- `AttachmentTitle`: `filename`, source symbol name, or URL basename
- `AttachmentDescription`: mime, source path, symbol name, or resource client
- `AttachmentTrigger` if URL/source preview is supported later

Conditions:
- Image MIME can use `AttachmentMedia variant="image"` if a renderable URL is safe.
- Source text should not be dumped inline by default; provide preview later.

#### `ToolPart`

Fields:
- `type: "tool"`
- `callID`, `tool`
- `state.status`: `pending`, `running`, `completed`, `error`
- `state.input`
- completed state has `output`, `title`, `metadata`, `time`, `attachments?`
- error state has `error`

Usefulness:
- Very high for OpenCode. Tool calls explain what the assistant did: read files, searched, edited, ran commands, asked permissions, etc.

UI mapping by status:
- `pending`
  - `Marker role="status"`
  - Text: `Queued {tool}`
- `running`
  - `Marker role="status"`
  - Text: `Running {title || tool}`
- `completed`
  - Compact `Marker variant="border"` for one-line summary
  - Optional `Bubble variant="outline"` for output summary
  - `AttachmentGroup` for `state.attachments`
- `error`
  - `Bubble variant="destructive"` or `Marker variant="border"`
  - Text: `{tool} failed: {error}`

Conditions:
- Do not dump large `input` or `output` by default.
- Show tool `title` first, fallback to `tool`.
- Long output should be collapsed/summarized in later implementation.
- Tool attachments map through the same `FilePart` attachment UI.

#### `SubtaskPart`

Fields:
- `type: "subtask"`
- `prompt`, `description`, `agent`, `model?`, `command?`

Usefulness:
- Medium. Useful for showing delegation/subagent activity.

UI mapping:
- `Marker variant="border"`
- Text: `Subtask: {description}`
- Include agent/model in `MarkerContent` or a muted detail line.

Conditions:
- If command exists, include compact command hint.
- Do not make it a full chat bubble unless the subtask has user-visible output.

#### `StepStartPart`

Fields:
- `type: "step-start"`
- `snapshot?`

Usefulness:
- Medium. Useful for activity timeline and streaming/progress.

UI mapping:
- `Marker role="status"`
- Text: `Started step` or a more specific label if inferable from adjacent parts.

Conditions:
- If too noisy, hide in completed historical transcripts and only show during active streaming.

#### `StepFinishPart`

Fields:
- `type: "step-finish"`
- `reason`, `snapshot?`
- `cost`, `tokens`

Usefulness:
- Medium. Useful for status and cost/tokens, but not primary conversation content.

UI mapping:
- `Marker`
- Text: `Finished: {reason}`
- Optional `MessageFooter` metadata for cost/tokens.

Conditions:
- Show compactly.
- Hide token/cost details behind a future detail toggle unless the UI is in developer mode.

#### `PatchPart`

Fields:
- `type: "patch"`
- `hash`
- `files: string[]`

Usefulness:
- High for a code assistant, medium for a personal assistant. Important when code/file changes happen.

UI mapping:
- `AttachmentGroup` containing one `Attachment` per changed file, or one summary attachment.
- `Bubble variant="outline"` if showing a patch summary.
- `Marker variant="border"`: `Changed {files.length} files`.

Conditions:
- If one file, show file attachment directly.
- If many files, show summary first and file list as attachments.

#### `SnapshotPart`

Fields:
- `type: "snapshot"`
- `snapshot`

Usefulness:
- Low for normal users, medium for recovery/debugging.

UI mapping:
- `Marker variant="separator"` or muted `Marker`
- Text: `Snapshot saved`

Conditions:
- Do not show raw snapshot IDs prominently.
- Consider hiding by default in a personal assistant UI.

#### `AgentPart`

Fields:
- `type: "agent"`
- `name`
- `source?`

Usefulness:
- Medium. Shows agent selection/switches.

UI mapping:
- `Marker`
- Text: `Agent: {name}` or `Switched to {name}`

Conditions:
- Show when agent changes or differs from the message-level agent.

#### `RetryPart`

Fields:
- `type: "retry"`
- `attempt`
- `error`
- `time.created`

Usefulness:
- Medium to high when errors occur.

UI mapping:
- `Marker variant="border"`
- Text: `Retry {attempt}: {error.data.message}`
- Use destructive styling only if the final result failed.

Conditions:
- Group repeated retries if noisy.

#### `CompactionPart`

Fields:
- `type: "compaction"`
- `auto`
- `overflow?`
- `tail_start_id?`

Usefulness:
- Low to medium. It explains why earlier context may be summarized.

UI mapping:
- `Marker variant="separator"`
- Text: `Conversation compacted` or `Context compacted automatically`

Conditions:
- Show as a timeline separator, not a chat bubble.

## Recommended Display Priority For Personal Assistant UI

1. Primary readable conversation
   - User `text`
   - Assistant `text`
   - Assistant `error`
2. Useful context and artifacts
   - `file`
   - `patch`
   - completed/error `tool`
3. Live/status transparency
   - pending/running `tool`
   - `step-start`, `step-finish`
   - `retry`
4. Advanced/debug context
   - `reasoning`
   - `agent`
   - `subtask`
   - `snapshot`
   - `compaction`
   - token/cost/path metadata

## Recommended Render Model

Keep one transcript row per OpenCode message entry, with nested rendering of parts:

```ts
type OpenCodeChatEntry = {
  info: Message
  parts: Part[]
}

type RenderedChatRow = {
  id: string
  role: "user" | "assistant"
  sender: string
  time?: string
  textParts: string[]
  reasoningParts: string[]
  fileParts: FilePart[]
  toolParts: ToolPart[]
  patchParts: PatchPart[]
  markerParts: Array<
    | StepStartPart
    | StepFinishPart
    | SnapshotPart
    | AgentPart
    | RetryPart
    | CompactionPart
    | SubtaskPart
  >
  error?: AssistantMessage["error"]
  metadata?: {
    agent?: string
    model?: string
    provider?: string
    cost?: number
    tokens?: AssistantMessage["tokens"]
  }
}
```

Rendering order inside a row:

1. Header: sender/time/model if enabled.
2. Markers that logically happen before visible text, such as agent switch or step start.
3. Main text bubble, if text exists.
4. Reasoning block, if enabled and present.
5. Tool cards/markers in part order.
6. Attachments and patch summaries.
7. Footer metadata and completion state.

## Implementation Plan

### Phase 1: Replace Dummy Rows With Query State

Modify only `webapp/src/routes/chats/$chatId.tsx` initially.

- Remove `void chatLogQuery`.
- Add rendering branches:
  - loading: `Marker role="status"` with `Loading chat...`
  - error: `Bubble variant="destructive"` or `Marker` with error message
  - empty: `Marker variant="separator"` with `No messages yet`
  - success: map OpenCode entries to rendered rows
- Keep `MessageScrollerProvider` and `MessageScrollerItem` structure.
- Keep user messages as `scrollAnchor` rows.

### Phase 2: Add Pure Mapping Helpers

Create local helpers at the bottom of `webapp/src/routes/chats/$chatId.tsx` first. Move to a separate module only if the route becomes too large.

Recommended helpers:

```ts
function mapOpenCodeChatLog(entries: ChatLogEntry[]): RenderedChatRow[]
function getMessageSender(info: Message): string
function getMessageTime(info: Message): string
function filePartToAttachment(part: FilePart): ChatAttachment
function toolPartToDisplay(part: ToolPart): ToolDisplay
function partId(entryId: string, part: Part, index: number): string
```

Implementation notes:
- Use discriminated unions on `part.type` and `info.role`.
- Add an exhaustive `switch` with fallback marker for unknown future part types.
- Preserve backend order of entries and parts.
- Avoid rendering raw large JSON by default.

### Phase 3: Build Dedicated Part Renderers

Still in `webapp/src/routes/chats/$chatId.tsx` unless it becomes unwieldy.

Add focused components:

- `OpenCodeMessageRow`
- `OpenCodeTextBubble`
- `OpenCodeReasoningBlock`
- `OpenCodeToolPart`
- `OpenCodeFileAttachments`
- `OpenCodePatchSummary`
- `OpenCodeMarkerPart`

Use existing shadcn primitives only at first. Do not add new component dependencies unless needed.

### Phase 4: Personal Assistant Polish

After the first real-data render works:

- Collapse or hide noisy debug-only parts.
- Summarize tools by title/status rather than raw input/output.
- Add better file icons by MIME/path.
- Add optional metadata mode for tokens/cost/model.
- Add streaming updates later using OpenCode events, but do not attempt that in the first render pass.

## Conditional UI Rules

- User text exists:
  - `Message align="end"`, `Bubble variant="default"`
- Assistant text exists:
  - `Message align="start"`, `Bubble variant="secondary"`
- Assistant error exists:
  - `Bubble variant="destructive"`
- File parts exist:
  - `AttachmentGroup` under the relevant message bubble
- Tool pending/running:
  - `Marker role="status"`
- Tool completed:
  - `Marker variant="border"` plus optional `Bubble variant="outline"` for concise output
- Tool error:
  - `Bubble variant="destructive"` or destructive marker text
- Patch part exists:
  - `Marker variant="border"` and file attachments
- Reasoning exists:
  - hidden/collapsed by default or `Bubble variant="ghost"` if explicitly enabled
- Snapshot/compaction exists:
  - subtle `Marker variant="separator"`
- No visible parts:
  - fallback `Marker` describing the message/part type

## Edge Cases

- `chatLogQuery.data` is `undefined` despite success.
- Session not found or unauthorized.
- Empty message list.
- Message with only non-text parts.
- Multiple text parts in one message.
- Very large tool output or file source text.
- Unknown future part type from OpenCode.
- Assistant message has `error` plus partial text/tool parts.
- Timestamps missing or incomplete.
- Duplicate part IDs.
- User prompt includes files but no text.
- Tool attachments reuse `FilePart` shape.

- Do not add streaming/event subscription yet.
- Do not build a full diff viewer yet.
- Do not render raw large tool JSON inline by default.
- Do not add new shadcn components unless an implementation blocker appears.
