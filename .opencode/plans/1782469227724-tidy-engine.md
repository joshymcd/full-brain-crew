# Unified Docker + Caddy Deployment Plan

## Goal

Move the project to one production deployable container that works on Railway, any Docker host, and local Docker Compose.

The public HTTP entrypoint should be Caddy. Caddy will:

- Serve the built React webapp at `/`.
- Proxy OpenCode SDK/API traffic at `/api/*` to the private OpenCode server.
- Proxy the original OpenCode web UI at `/admin/*` for low-level/debug/admin access.

OpenCode itself should no longer be the public listener. It should run privately on an internal port such as `127.0.0.1:8080`.

## Recommended Architecture

```txt
Browser
  -> Railway/public Docker host $PORT
    -> Caddy
      -> /             static webapp dist
      -> /api/*        reverse proxy to OpenCode on 127.0.0.1:8080
      -> /admin/*      reverse proxy to OpenCode web UI on 127.0.0.1:8080
```

This removes the need for a separate frontend Railway service, cross-origin browser API calls, Vite preview in production, and build-time backend public URLs.

## Key Design Decisions

- Keep a single root `Dockerfile` as the production image.
- Add Caddy to the production image and make it the only process bound to Railway `$PORT`.
- Build `webapp/` during Docker image build and copy the output into a static directory such as `/usr/share/caddy`.
- Replace `exec opencode web --port "$PORT"` with a supervised startup that runs OpenCode on a fixed internal port and Caddy in the foreground.
- Change the frontend OpenCode SDK base URL fallback from `http://localhost:8080/` to same-origin `/api/`.
- Keep the frontend Basic Auth behavior: user enters the OpenCode username/password in the app, and SDK calls send the `Authorization: Basic ...` header through Caddy to OpenCode.
- Keep `/admin/` as the path for the original OpenCode web UI.

## Implementation Steps

1. Update The Root Dockerfile

- Keep the current `node:22-trixie-slim` base unless image size becomes a real issue.
- Install `caddy` alongside existing runtime dependencies.
- Enable Corepack.
- Copy only webapp dependency manifests first for Docker layer caching:
  - `webapp/package.json`
  - `webapp/pnpm-lock.yaml`
  - `webapp/pnpm-workspace.yaml`
- Run `pnpm install --frozen-lockfile` inside a webapp build stage or build section.
- Copy `webapp/` and run `pnpm run build`.
- Copy the built frontend output into a static-serving directory, likely `/usr/share/caddy`.
- Copy `deploy/Caddyfile` into `/etc/caddy/Caddyfile`.
- Preserve the existing OpenCode, Obsidian, Google Workspace, crew installer, `/vault`, and runtime script setup.

2. Add `webapp/.dockerignore`

- Exclude local-only/build artifacts so Docker does not copy them into the image:
  - `node_modules`
  - `dist`
  - `.vite`
  - `.env`
  - `.env.*`
  - `*.log`
- Re-include `.env.example` if needed.

3. Add Caddy Configuration

Create `deploy/Caddyfile` with a public listener on Railway/Docker `$PORT` and path-based routing.

Recommended shape:

```caddyfile
:{${PORT:8080}}

handle_path /api/* {
  reverse_proxy 127.0.0.1:8080
}

handle_path /admin/* {
  reverse_proxy 127.0.0.1:8080
}

handle {
  root * /usr/share/caddy
  try_files {path} /index.html
  file_server
}
```

Validate whether `handle_path` prefix stripping works for both OpenCode SDK endpoints and the OpenCode web UI. If the OpenCode UI requires root-relative assets or websocket paths, adjust the admin route to either:

- Use a subdomain instead of `/admin`, or
- Use selective proxy routes for OpenCode-owned asset/websocket paths, or
- Use Caddy rewrite/header handling if practical.

Start with `/admin/` because it matches the desired UX, but verify it manually.

4. Update Entrypoint Process Supervision

Modify `scripts/entrypoint.sh` so it still performs all current setup:

- Select and start sync backend.
- Install/update Full Brain Crew into `$WORKSPACE_PATH`.
- Configure GWS credentials.
- Ensure workspace git metadata.
- `cd "$WORKSPACE_PATH"` before starting OpenCode.

Then start OpenCode privately and Caddy publicly:

```bash
PUBLIC_PORT="${PORT:-8080}"
OPENCODE_INTERNAL_HOST="${OPENCODE_INTERNAL_HOST:-127.0.0.1}"
OPENCODE_INTERNAL_PORT="${OPENCODE_INTERNAL_PORT:-8080}"

opencode web --port "${OPENCODE_INTERNAL_PORT}" --hostname "${OPENCODE_INTERNAL_HOST}" &
OPENCODE_PID="$!"

trap 'kill "${OPENCODE_PID}" 2>/dev/null || true' INT TERM EXIT

caddy run --config /etc/caddy/Caddyfile --adapter caddyfile &
CADDY_PID="$!"

wait -n "${OPENCODE_PID}" "${CADDY_PID}"
```

Implementation detail: avoid using the same variable for public and internal ports. Railway owns `PORT`; OpenCode should use `OPENCODE_INTERNAL_PORT`.

If `wait -n` is used, make sure the script exits nonzero when either critical process dies and cleans up the other process.

5. Update Frontend OpenCode URL

Change `webapp/src/opencode-client.tsx`:

```ts
const OPENCODE_SERVER_URL = import.meta.env.VITE_OPENCODE_SERVER_URL ?? "/api/";
```

Keep support for `VITE_OPENCODE_SERVER_URL` as an override for unusual deployments or local development.

6. Update Docker Compose

Replace the current production-ish `app` plus dev `webapp` shape with a clear production default:

```yaml
services:
  app:
    build: .
    ports:
      - "${APP_PORT:-8080}:8080"
    env_file:
      - path: .env
        required: false
    volumes:
      - ./.vault:/vault
```

Do not map OpenCode's internal port publicly in the production compose path.

Optionally add a separate `docker-compose.dev.yml` later for hot-reload development:

- OpenCode service on `8080`.
- Webapp Vite dev service on `3000`.
- Vite dev proxy for `/api` and `/admin`, or a Caddy dev proxy.

7. Update Railway Config

Keep root `railway.json` as the source of truth:

```json
{
  "build": {
    "builder": "DOCKERFILE",
    "dockerfilePath": "Dockerfile"
  },
  "deploy": {
    "restartPolicyType": "ON_FAILURE",
    "restartPolicyMaxRetries": 3
  }
}
```

No separate frontend Railway service should be required after this migration.

8. Update Environment Documentation

Update `.env.example` and `README.md` to explain:

- Public app is served by Caddy on Railway `$PORT` or local `${APP_PORT:-8080}`.
- The custom webapp is at `/`.
- OpenCode fallback/admin UI is at `/admin/`.
- OpenCode SDK/API proxy is at `/api/`.
- `VITE_OPENCODE_SERVER_URL` usually does not need to be set in production anymore.
- `OPENCODE_PORT` should be replaced or clarified because the OpenCode port is now internal. Prefer `APP_PORT` for host mapping and `OPENCODE_INTERNAL_PORT` for internal runtime.

## Risks And Edge Cases

- OpenCode web UI may not work cleanly under `/admin/` if it assumes it owns `/` for static assets or websocket paths. Verify this early. If it fails, consider exposing OpenCode UI under a separate Caddy host when a custom domain is available, or use `/admin` only for API-compatible pages if possible.
- OpenCode SDK endpoints may or may not tolerate prefix stripping. Use `handle_path /api/*` first because it strips `/api` before proxying.
- Websocket/SSE endpoints may need explicit Caddy support. Caddy reverse proxy usually handles websocket upgrades automatically, but verify with real OpenCode chat/session interactions.
- Basic Auth should be passed through unchanged. Do not add Caddy `basicauth` initially, because OpenCode already enforces its own credentials and the frontend injects that header.
- Process supervision must ensure the container exits if either OpenCode or Caddy dies.

3. Local Container Smoke Test

- Run:
  - `docker run --rm -p 8080:8080 --env-file .env full-brain-crew:unified`
- Open:
  - `http://localhost:8080/` for the new webapp.
  - `http://localhost:8080/admin/` for the original OpenCode UI.
- In the new webapp, enter the same OpenCode username/password and verify SDK calls succeed through `/api/`.

4. Docker Compose Test

- Run:
  - `docker compose up --build`
- Verify the same URLs using `${APP_PORT:-8080}`.

5. Railway Test

- Deploy the root Railway service only.
- Verify the public Railway URL serves the new webapp at `/`.
- Verify `/admin/` reaches the original OpenCode UI.
- Verify login from the new app and OpenCode SDK queries work without CORS errors.

6. Failure Behavior

- Confirm container exits/restarts if Caddy fails.
- Confirm container exits/restarts if OpenCode fails.
- Confirm sync failures still respect `SYNC_REQUIRED`.
